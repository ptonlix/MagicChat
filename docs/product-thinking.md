# 产品思路

## 一句话想法

> 做一个 AI 原生的企业工作入口：让每个人拥有自己的工作助理，自动接收消息、整理话题、提取任务、执行事务，并在权限边界内代表用户协作。

## 背景

- 为什么值得做：钉钉、飞书、企业微信、Slack、Teams 等企业协作套件已经被广泛采用，但它们的核心形态仍然是传统聊天、文档、审批和工作流。AI 时代的工作方式应该被重新设计。
- 变化点：AI 已经改变了用户对软件的期待。用户不只希望软件展示信息，而是希望软件理解上下文、处理消息、推进任务、调用工具、在不确定时升级给人。
- 现有替代方案：钉钉、飞书、企业微信、Slack、Teams，以及聊天、文档、会议、审批、项目管理、邮件工具的拼接组合。
- 更相关的参考对象：Kylon、Bloome、Glue、Raft、FloatIM、Ano、Den、Tanka 等，它们分别覆盖人机协作、Agent 原生 IM、AI workspace、话题式协作、任务式 Agent 和 assistant-first 工作入口。Teamily 只作为边界参考，不作为核心对标，因为它明显偏社交网络。详见 [competitor-analysis.md](competitor-analysis.md)。
- 更完整的赛道图谱：见 [ai-native-enterprise-im-landscape.md](ai-native-enterprise-im-landscape.md) 和 [ai-native-enterprise-im-project-introductions.md](ai-native-enterprise-im-project-introductions.md)。

## 目标用户

- 主要用户：每天被消息、邮件、会议、文档、审批、任务和跨工具协作打断的知识工作者。
- 次要用户：需要工作自动推进、不想靠人工催办和翻群聊追上下文的管理者。
- 购买决策者：中小团队创始人、业务负责人、IT/数字化负责人，以及重视效率和安全边界的企业管理层。

## 问题

- 核心痛点：现有企业协作产品仍然要求人亲自阅读、分流、回复和跟进大量消息。
- 更深层问题：主流平台不是 AI 原生的。它们通常只是把 AI 功能加到传统 IM、文档和审批系统上，而不是围绕 AI 重新设计工作入口和协作结构。
- Agent 身份问题：很多平台没有开放友好的接口让 AI Agent 像成员一样参与工作。机器人往往只能被 @，不能自然接收消息、观察上下文、后台执行任务或主动发起对话。
- 消息结构问题：传统群组只是“人群容器”，不是“工作容器”。多人在一个群里聊天时，重要话题、决策、任务和待跟进事项很容易被时间流淹没。
- 当前工作流：人仍然是消息、协调、跟进、路由的默认处理者。AI 只有在人显式调用时才提供帮助，通常不能拥有任务，也不能判断何时升级给本人。
- 现有方案不足：AI 被当成命令工具，而不是拥有身份、权限、记忆、行动接口和责任边界的协作者。

## 方案方向

- 核心价值：一个 AI 原生工作平台。每个人都有个人工作助理，Agent 可以像组织成员一样参与协作，而不是被动机器人。
- 部署原则：产品采用独立部署形态，不以 SaaS 为默认形态。客户先安装一个服务端，在服务端完成组织、用户、外部账号、模型、Agent、权限和集成配置；员工再通过桌面端、移动端或 Web 客户端登录使用。详见 [deployment-model.md](deployment-model.md)。
- 关键流程：消息和工作请求先进入用户的助理。助理负责理解、总结、分流、草拟、回复、建任务、约时间、跟进、后台执行；处理不了或风险较高时再升级给本人。
- 三层产品模式：
  1. 个人 AI 助理：作为用户的日常助理，用户可以随时向它提问、委托任务、让它处理消息和管理 todo。
  2. 团队聊天工具：作为团队内的人与人、人和 Agent、Agent 和 Agent 协作空间，支持成员沟通、话题组织、任务流转和上下文沉淀。
  3. Agent 平台：每个 Agent 都可以作为一个独立应用存在，拥有自己的能力、入口、权限和对话界面；用户或助理可以调用它们完成特定工作。
- 助理调度原则：用户的个人 AI 助理是“总控入口”。它可以帮用户完成产品内的所有操作，例如控制其他 Agent 干活、给某些人发消息、创建或更新 todo、管理话题、查询上下文、发起审批或升级给本人确认。
- Agent 身份原则：Agent 应该拥有接近人的组织身份，可以接收消息、参与群聊、关注话题、拥有任务、后台执行，并在权限允许时主动发起对话。
- 助手优先原则：产品默认逻辑是“助理先处理，人再审核或介入”，而不是“人先被打断，AI 再补救”。
- 集成原则：产品应连接外部消息平台，如邮件、其他 IM、日历和企业工具，让助理成为统一工作入口。
- 外部账号连接原则：用户可以配置自己的邮件、钉钉、飞书、企业微信、Slack、Teams 等外部账号。授权后，助理可以统一接收、搜索、归类和处理这些渠道里的消息。
- 跨渠道代执行原则：助理不仅能读取外部消息，也应能在授权范围内执行动作，例如代发邮件、回复 IM、转发信息、创建日历邀请、同步 todo、把外部消息归并到产品内的话题或任务。
- 渠道边界原则：外部渠道的身份、语气、接收者、可见范围和合规要求不同。助理在代发邮件、回复钉钉/飞书消息等出站动作前，应根据渠道、对象、内容敏感度和用户设置决定自动发送、草拟待确认或禁止执行。
- 话题优先原则：群组表达人、成员关系和权限；话题表达具体事务、问题、请求、事件、决策或项目。讨论应围绕话题展开，让上下文、决策、任务和 Agent 行动可以被恢复和追踪。
- 任务提取原则：进入系统的消息应自动分析请求、承诺、待跟进、风险、截止时间和决策，提取候选任务，并在任务面板中管理。
- Agent 执行原则：任务不只是提醒。助理或团队 Agent 应能执行任务的一部分，例如草拟回复、查资料、约会议、建文档、更新系统、追问信息或调用工具。
- 日常助手原则：用户应能经常和自己的助理聊天，提问、委托工作、要求执行任务。如果有人找用户但用户没有及时看到，助理可以根据权限和置信度分流、草拟或回复。
- 边界与隐私原则：助理不能靠“自觉”判断边界。任务边界、隐私作用域、可用上下文、出站动作规则和审批要求必须是产品的一等机制。
- 敏感数据模型：HR 薪资数据是典型例子。同一条数据能否读取、能否用于推理、能否说出口、能否触发动作，取决于角色、目的、接收者和审批状态。详见 [privacy-permission-model.md](privacy-permission-model.md)。

## 差异化

- 推荐定位：assistant-first enterprise work inbox，也就是“助理优先的企业工作入口”。它不是社交 AI 网络，也不只是 AI 群聊。
- 核心承诺：工作先找用户的 AI 助理；助理负责结构化、处理、执行或升级，并始终遵守权限边界。
- 用户为什么会用：减少亲自读消息、分流、回复、催办和翻上下文的负担。
- 难复制优势：Agent 身份、权限、消息结构、话题、任务、记忆、工具调用和审计从平台层一起设计，而不是传统 IM 后加机器人。
- 市场切口：不要从“替代飞书/钉钉/Slack 全套功能”开始，而是先做跨邮件、IM、日历、任务和企业工具的 assistant-first 工作入口。
- 竞品启发：“Agent 进群聊”已经是活跃赛道。真正更尖锐的问题是：助理能否在消息到达人之前先拥有工作处理权。
- 非目标：不做朋友、家庭、社区、内容发现或消费社交网络。产品必须锚定企业工作、责任、权限、执行和可审计性。
- 结构启发：借鉴 Glue 的话题/线程形态，避免把一个巨大的群时间流作为默认工作单元。每个话题应是持久、可访问的工作空间，能承载消息、决策、任务、文件、总结、Agent 执行和升级状态。
- 任务启发：借鉴 Raft 的任务模式。产品应有专门任务面板，集中展示 AI 提取任务、用户创建任务、Agent 拥有任务、被委派任务和等待人工审批的任务。
- 部署差异化：私有化/独立部署更适合企业授权邮件、IM、HR、财务、客户等敏感数据给 AI 助手处理。相比纯 SaaS，更容易建立数据边界、审计、模型选择和企业集成信任。

## 需要突出的产品特色

1. 助手优先的工作入口：消息、邮件、@、会议邀请、任务请求先由个人助理处理。
2. 话题优先的沟通结构：群组负责人和权限，话题负责上下文、决策、文件、总结和 Agent 行动。
3. AI 自动提取任务：助理从消息中识别请求、承诺、截止时间、待跟进和风险，形成任务面板。
4. Agent 能执行，而不只是总结：助理和团队 Agent 可以草拟、研究、约时间、建文档、更新工具、追问信息和后台运行。
5. 隐私感知的委托：所有动作都遵守接收者、目的、任务边界、敏感数据规则和审批要求。
6. 跨渠道控制层：产品先连接已有 IM、邮件、日历和企业工具，不要求第一天完整替换现有平台。
7. Agent 作为组织成员：Agent 有身份、角色、权限、记忆、任务归属、审计日志和升级规则。
8. 外部账号统一接管：用户配置邮件、钉钉、飞书等账号后，助理可以统一收消息、提任务、整理话题，并按权限代表用户回复或发起外部沟通。
9. 独立部署与企业控制：企业自己安装服务端，配置模型、连接器、权限、审计和数据存储；用户通过客户端登录使用。

## 商业模式

- 可能计费方式：按私有化部署授权、用户席位、助理席位、Agent 席位、企业连接器、模型网关、安全审计能力和支持服务分层。
- 分发渠道：AI 原生团队、工程团队、业务运营团队、HR/招聘/客服/销售等高消息密度团队。
- 验证需求的指标：未读消息减少量、自动提取任务数量、人工回复减少量、任务完成率、升级准确率、用户节省时间、敏感数据误用率。

## 风险与开放问题

- 用户信任风险：用户可能不信任助理代表自己回复、承诺或主动发起对话。
- 市场风险：Slack、Teams、飞书、钉钉、OpenAI、Anthropic、Microsoft、Atlassian 等大平台会持续加入 Agent 能力。
- 技术风险：如何安全地给 Agent 足够上下文和行动能力，同时避免隐私、合规和安全问题。
- 边界风险：助理可能做得太多、在错误场景使用隐私信息、回复错误对象，或在原始任务边界结束后继续行动。
- 集成风险：连接外部邮件和 IM 平台会遇到权限、速率限制、企业审批和平台合规问题。
- 部署风险：独立部署会增加安装、升级、运维、故障排查、模型配置、数据备份和企业网络适配成本。

## 下一步讨论

- 第一个要验证的假设：用户是否愿意让助理先接住消息，而不是所有消息直接打断本人。
- 最小可用原型：先做管理员创建用户、通讯录、内置 AI 助手对话、用户一对一聊天。详见 [mvp-requirements.md](mvp-requirements.md)。
- 需要访谈的人：消息密度高的管理者、HR、招聘、客服、销售、运营、工程负责人。
- 下一步决策：先做个人 AI 工作收件箱，还是先做团队话题 + 任务协作空间。
